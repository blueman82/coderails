### SO-71: A selector that can silently match NOTHING turns a gate into a no-op — treat an empty lookup result as a REFUSAL, not as "nothing to do," especially across a type boundary a scripting language erases

- **Created:** 2026-07-24
- **Failure mode:** `scripts/post_evals.sh::smoke_run` looked up each eval's `cmd`/`negative_control` with `jq --arg id "$id" '.evals[] | select(.id == $id)'`. `--arg` always binds a shell STRING, so when an eval's `id` was authored as a JSON number, `select(.id == $id)` never matched anything — jq's `==` does not coerce number-vs-string. The lookup returned empty, `cmd`/`nc` became empty strings, nothing executed, and the function wrote `smoke: {cmd_exit: null, ...}` and still returned exit 0. The gate whose entire purpose is "a neutral script computes the evidence so the agent can't just claim it" recorded evidence for a command that was never run, and reported success while doing so. The defect was found incidentally — an eval-authoring worker hit the null-smoke symptom live while building artifacts for an unrelated PR in the same session, and the mechanism was verified from source afterward.
- **Lesson:** When a gate's evidence-gathering step depends on a lookup/selector (a `jq select`, a map key, a query filter) succeeding, an empty result must be a hard failure, never silently treated as "no matching work." This is sharpest at a type boundary a scripting language quietly erases: shell variables are always strings, so any `--arg`-bound jq comparison against a schema field that COULD be authored as a non-string (a bare number, a bool) needs either an explicit type guard before the lookup, or `--argjson`/index-based access that can't silently mismatch. Before trusting a selector-based gate, ask what it does on zero matches — if the answer is "returns cleanly," that is the fail-open to close.
- **Evidence:** session `4601b8c4-27d9-4460-8960-d61f090ac33a`, PR #294 (`fix/post-evals-numeric-id`, head `35dabcc60715964a6218ddd9de86b8c3c8f85198`). Fix: a type guard added at the top of `smoke_run` (`scripts/post_evals.sh`, +13 lines) refusing any scripted eval with a non-string `id`, +20 lines `hooks/scripts/tests/post_evals.test.sh`. Verified this session: numeric id → `smoke-run` exit 1, stderr "a scripted eval has a non-string id (schema requires id to be a string) — refusing."; string id → exit 0, real observed `{"cmd_exit":0,"negative_control_exit":1,...}` written; full suite `bash hooks/scripts/tests/post_evals.test.sh` → PASS. Sibling fact worth noting for future readers of this file: the same script's check 10 (`validate_smoke_execution`, `scripts/post_evals.sh:415,423-424`) was NOT vulnerable to this defect, but for a reason unrelated to correctness — it iterates `.evals[$idx]` by array INDEX rather than the `select(.id == $id)` string-bound pattern used at the other call sites (lines 258, 272-273, 719-720 now-guarded, 743, 1043, 1050-1053); those other sites were left unchanged in the fix because their existing shape checks already fail closed on a numeric id, just with a less clear error message.
