## Promoted lessons

### A green test suite is not evidence the gate works — only a negative control run against the code that HAD the bug is

- **Failure mode:** Real defects in infrastructure ship while a test suite reports green because the suite was authored by the same process that wrote the code, so it encodes the same blind spot. A green suite cannot catch bugs the author couldn't imagine. Worse, a broken fix can also pass 39/39 green and leave a regression undetected until an independent reviewer exercises it.
- **Lesson:** For any gate, guard, or check, the only evidence that counts is the **negative control** — run the check against the code that actually had the bug and watch it **fail**. "Tests pass" answers "is the code self-consistent?", never "does the gate do its job?" A control that has never failed is decoration. Two corollaries: (1) **Verify the control reverted** — `git stash push` on a committed file is a silent no-op; use `git checkout <base-ref> -- <file>` against an explicit ref and confirm the revert took before trusting any control. (2) **Evals authored post-hoc share the implementer's oracle** — derive every assertion from the bug report's goal state, never shell out to the repo's own test suites. Reviewers' independent exploit inputs are worth more than any assertion the author could invent.
- **Evidence:** Recurred 7 times across 7 distinct sessions (fb2bfec4, 2e82b342, 007e525b, bc0e6ad9, 75fa2e68, f876b596, 0692aeab). Session fb2bfec4 had two real bugs live at 39/39 green on main; first fix shipped a worse defect also at 39/39; only independent review caught both. Loop 1cc4c7d0 had five frozen evals with correct intent but wrong observation mechanisms, all invisible at freeze because the smoke-run only saw the fail side (pre-build). Loop 75fa2e68 proved the sharpened rule: a check only observed failing has not been shown to discriminate — exercise the formula against a **synthetic pass-shaped input** too at freeze time.
- **Promotion date:** 2026-07-20

### A frozen, blind-authored check can itself be defective — validate each scripted check's discriminating power before freezing

- **Failure mode:** A frozen check that has never produced both a pass and a fail is not yet a check. An awk formula exiting 1 unconditionally, a proof cmd whose grep pattern never occurs in its input, a process substitution unreadable in a non-bash shell — all are silent defects invisible at freeze if the check is only observed failing (pre-build code, everything fails for the intended reason). Checks with correct intent but wrong observation mechanisms can ship and leave suites vacuous.
- **Lesson:** At freeze time, run each scripted `cmd` and `negative_control` once for real in the actual execution shell, and smoke-test both environment plumbing and formula correctness. If the check carries optional `fixtures` (synthetic pass-shaped and fail-shaped inputs), validate the formula can discriminate — require opposite exit codes from good vs bad fixtures. A check observed only failing is half-validated at best; exercise both sides at freeze. Amendment path (not rewrite): recorded reason, assertion unchanged, fresh re-grade if a verdict exists. For agent-run evals, smoke-test the grader's prompt structure and confirm it receives only the artifact references, never the implementation conversation — the author is the least able to see their own shims.
- **Evidence:** Recurred 4 times across multiple sessions (8b69e779, bc0e6ad9, 75fa2e68, 1cc4c7d0, 5f525531). Loop 8b69e779 had four frozen checks broken in ways unrelated to the artifact they judged; loop bc0e6ad9 had three frozen evals whose `cmd` had never been run and would fail on first execution (reporter flag doesn't exist, npm workspaces mismatch, gate self-blocks). Loop 1cc4c7d0 had five instances with intent-correct-mechanism-wrong, invisible at freeze — the two-sided smoke-run would have caught all five. Loop 5f525531 froze an eval asserting a regression lock would ship in one file when the fix correctly chose another; smoke-run on synthetic fixtures would have discriminated.
- **Promotion date:** 2026-07-20

### The Bash tool's working directory drifts between calls to whichever sibling worktree was most recently entered — anchor every command

- **Failure mode:** In multi-worktree sessions, a shell's cwd silently resets between tool calls to a different worktree of the same repo (whichever was most recently entered). A worker's test run can execute against the wrong repo's code, test assertions pass with wrong cardinality, and git status comes back suspiciously clean. The drift is transparent to the caller unless the output is cardinality-sensitive (test counts, assertion totals, list lengths) or the cwd is visibly wrong.
- **Lesson:** Never rely on cwd persistence between Bash calls in multi-worktree loops. Start every command with an explicit `cd <absolute worktree path> &&`, use `git -C <absolute path>`, or run git/file commands with absolute paths throughout. Treat known-cardinality outputs (test counts, assertion totals) as identity checks — a mismatch means you may be in the wrong tree. Edits with absolute file paths are immune to cwd drift; a probe pinned to a coordinate the repo can move under (line numbers via `sed -n N`, whole-file greps, working-tree path) is a stale instrument.
- **Evidence:** Recurred 2 times across 5 distinct sessions (0767967d loop ordinal 1; 8a43eccc; eafadd31; 47859e78; b5fe4e97). Session 0767967d had a worker's test run against the wrong repo (caught via assertion-count mismatch) and orchestrator shell reset to a sibling worktree across multiple turns; both cases re-anchored with explicit `cd <absolute>`. Loop 8b69e779 had orchestrator cwd drifted into the builder's worktree causing a foreign stop-hook to demand a commit of someone else's in-flight work.
- **Promotion date:** 2026-07-20

### A worker's "all N items done" report on a multi-item fix list is a per-item claim, not one claim — verify each item against the merged artifact

- **Failure mode:** A multi-item fix (N distinct items) gets a single "done" report that collapses N independent claims into one. At the merge boundary, re-reading origin shows the report was partial: the first "done" had the gate fix but not the price or sanitisation; a later "C fixed" had the guard but the numeric leaves were still wrong. Each report was two-thirds true because messages crossed in flight and "done" referred to an earlier, narrower ask. Worse, when a report is fully honest and the worker's self-verification genuinely passed, that is not evidence of completeness — it is evidence the worker's oracle and its scope agreed with each other. An independent reviewer with a different lens catches defects nobody thought to ask for.
- **Lesson:** When a fix list has N distinct items, apply Phase 12 (verify against merged state) **per item**, not per report. At merge boundary, grep/reproduce each item against the merged SHA by name (the price row, the sanitise call, the specific guard, the specific test). Expect "done" to mean "the last thing I understood you to ask," especially amid mailbox-crossing — multi-item fixes nearly always need one more verify-and-nudge round than the report implies. The cost of re-verifying each item is one grep; the cost of merging 2/3-done is a reopened crash class.
- **Evidence:** Recurred 2 times (47859e78 loop ordinal 1; f876b596 ordinal 2). Loop 47859e78 had three separate "done" reports each two-thirds true on a 4-item miner fix; each partial closed by re-reading origin and sending a per-item nudge. Loop f876b596 saw two independent reviewers find real defects (doc row contradicting its own sibling paragraph, test missing the actual-failure ordering) in PRs with honest, self-verified green reports whose scope simply hadn't asked the right question yet.
- **Promotion date:** 2026-07-20
